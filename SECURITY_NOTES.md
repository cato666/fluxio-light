# Security Notes

## Backend npm audit review

Date: 2026-05-27

Scope: `backend`, production dependencies only (`npm audit --omit=dev`).

## Applied low-risk changes

- Removed direct `uuid` dependency. The backend code does not import it; UUIDs are generated by PostgreSQL through Prisma schema defaults.
- Removed direct `multer@1.x` dependency. File uploads use Nest's `FileInterceptor`, and `@nestjs/platform-express` already provides the runtime multer dependency.
- Added npm overrides for compatible transitive patches:
  - `multer@2.1.1`
  - `qs@6.15.2`
  - `lodash@4.18.1`

## Pending by design

- `@nestjs/core` / `@nestjs/platform-express`: npm audit recommends `@nestjs/*@11.1.24`. This is a Nest major upgrade, so it was not applied in this task.
- `@nestjs/common` via `file-type`: Nest 10.4.22 pins `file-type@20.4.1`. The patched line is `file-type > 21.3.1`, which is a transitive major override. It was not forced because this project currently stays on Nest 10 and the safer fix should be evaluated together with the Nest 11 upgrade path.
- `@nestjs/config`: npm audit recommends `@nestjs/config@4.0.4`, a package major upgrade. A lodash override was applied instead to address the vulnerable transitive lodash without changing the Nest config package major.

## Recommended follow-up

- Plan a separate Nest 11 upgrade branch that updates the Nest packages together, runs the upload flow, auth flow, Prisma seed, Docker Compose, and dashboard smoke tests.
- Revisit `file-type` after that upgrade; avoid carrying a long-term major override for a Nest-pinned dependency unless tests cover upload validation paths.

## Pre-release audit

Date: 2026-05-31

Scope: local pilot readiness review before first external version.

### Completed

- Added `.gitignore` to keep `.env`, service-specific env files, local tunnel logs, uploads, build outputs and temporary files out of git.
- Verified `.env.example` has the same variable names as the local `.env`, without copying real secret values.
- Added a production startup guard: `NODE_ENV=production` now requires a non-placeholder `JWT_SECRET` and a concrete `FRONTEND_URL`.
- Verified Docker Compose starts backend, frontend and Postgres locally.
- Verified frontend responds on `http://localhost:5173`.
- Verified protected Admin endpoint rejects invalid JWT with `401`.
- Verified Kapso sandbox webhook script returns `200` and creates a processed message.

### Remaining before production

- Replace local Postgres credentials (`fluxio/fluxio`) with managed database credentials.
- Replace local/demo `JWT_SECRET`.
- Set `FRONTEND_URL`, `APP_URL`, `PUBLIC_STORAGE_BASE_URL` and `VITE_API_URL` to production URLs.
- Keep current Nest 10 npm audit findings as known moderate risk until a planned Nest 11 upgrade is tested.
- Do not run the seed against production data; it recreates demo data for `admin@fluxiolight.local`.
